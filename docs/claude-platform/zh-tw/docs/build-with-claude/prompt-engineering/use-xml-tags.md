# 使用 XML 標籤來結構化您的提示詞

了解如何使用 XML 標籤來改進提示詞的結構和清晰度，幫助 Claude 更準確地解析您的請求。

---

<Note>
While these tips apply broadly to all Claude models, you can find prompting tips specific to extended thinking models [here](/docs/en/build-with-claude/prompt-engineering/extended-thinking-tips).
</Note>

當您的提示詞涉及多個組件（如上下文、指令和範例）時，XML 標籤可以改變遊戲規則。它們幫助 Claude 更準確地解析您的提示詞，從而產生更高品質的輸出。

<Tip>**XML 提示**：使用 `<instructions>`、`<example>` 和 `<formatting>` 等標籤來清楚地分隔提示詞的不同部分。這可以防止 Claude 將指令與範例或上下文混淆。</Tip>

## 為什麼使用 XML 標籤？

- **清晰度：** 清楚地分隔提示詞的不同部分，並確保您的提示詞結構良好。
- **準確性：** 減少由 Claude 誤解提示詞部分而造成的錯誤。
- **靈活性：** 輕鬆找到、新增、移除或修改提示詞的部分，無需重寫所有內容。
- **可解析性：** 讓 Claude 在其輸出中使用 XML 標籤，可以更輕鬆地通過後處理提取其回應的特定部分。

<Note>沒有規範的「最佳」XML 標籤是 Claude 特別訓練過的，儘管我們建議您的標籤名稱應與其周圍的資訊相符。</Note>

***

## 標籤最佳實踐

1. **保持一致**：在整個提示詞中使用相同的標籤名稱，並在談論內容時參考這些標籤名稱（例如，`使用 <contract> 標籤中的合約...`）。
2. **嵌套標籤**：您應該為分層內容嵌套標籤 `<outer><inner></inner></outer>`。

<Tip>**進階使用者提示**：將 XML 標籤與其他技術結合，如多次提示（`<examples>`）或思維鏈（`<thinking>`、`<answer>`）。這會建立超級結構化、高效能的提示詞。</Tip>

### 範例

  <section title="範例：生成財務報告">

    沒有 XML 標籤時，Claude 會誤解任務，生成與所需結構或語調不符的報告。替換後，Claude 也可能誤解一個部分（如 Q1 報告範例）在哪裡結束，另一個部分在哪裡開始。

  <section title="範例：法律合約分析">

    沒有 XML 標籤時，Claude 的分析組織混亂且遺漏關鍵要點。使用標籤時，它提供結構化、徹底的分析，法律團隊可以採取行動。

    | 角色 | 無 XML 標籤 | 有 XML 標籤 |
    | ---- | ----------- | ------------- |
    | 使用者 | 分析此軟體授權協議以了解潛在風險和責任：\{\{CONTRACT\}\}。重點關注賠償、責任限制和知識產權所有權條款。另外，注意任何不尋常或令人擔憂的條款。這是我們的標準合約供參考：\{\{STANDARD_CONTRACT\}\}。為我們的法律團隊提供調查結果和建議的摘要。 | 分析此軟體授權協議以了解法律風險和責任。<br/><br/>我們是一家跨國企業，正在考慮將此協議用於我們的核心資料基礎設施。<br/><br/>\<agreement><br/>\{\{CONTRACT\}\}\</agreement><br/><br/>這是我們的標準合約供參考：<br/>\<standard_contract>\{\{STANDARD_CONTRACT\}\}\</standard_contract><br/><br/>\<instructions><br/>1. 分析這些條款：<br/>   - 賠償<br/>   - 責任限制<br/>   - 知識產權所有權<br/><br/>2. 注意不尋常或令人擔憂的條款。<br/><br/>3. 與我們的標準合約進行比較。<br/><br/>4. 在 \<findings> 標籤中總結調查結果。<br/><br/>5. 在 \<recommendations> 標籤中列出可行的建議。\</instructions> |
    | 助手 | 提供的軟體授權協議呈現了幾個令人擔憂的方面。賠償條款過於寬泛，可能使我們面臨重大責任。責任限制部分將損害賠償限制在僅 \$500，這對我們的企業級使用來說是不足的。關於知識產權，該協議授予供應商對我們建立或整合的任何修改的聯合所有權。<br/><br/>總之，此協議不符合我們的最佳利益。它使我們面臨不必要的風險，限制了我們在發生故障時的追索權，並損害了我們的知識產權。我建議重新協商這些條款或尋求替代供應商。 | \<findings><br/>1. 賠償（第 8 條）：<br/>   - 問題：過於寬泛。我們賠償供應商因我們使用而產生的「任何和所有索賠」。<br/>   - 風險：即使供應商疏忽或產品缺陷，也可能使我們承擔責任。<br/>   - 偏差：我們的標準將賠償限制為因我們違反協議而產生的索賠。<br/><br/>2. 責任限制（第 9 條）：<br/>   - 問題：將所有損害賠償限制在 \$500，包括直接損害。<br/>   - 風險：對於企業使用來說嚴重不足。關鍵故障可能造成數百萬美元的損失。<br/>   - 偏差：我們的標準允許直接損害賠償最高達支付費用的 12 個月。<br/><br/>3. 知識產權所有權（第 12 條）：<br/>   - 問題：授予供應商對我們修改/整合的聯合所有權。<br/>   - 風險：供應商可能使用或銷售我們的專有演算法或資料見解。<br/>   - 偏差：我們的標準保留對我們工作的完全所有權。<br/><br/>4. 不尋常的條款：<br/>   - 第 5.3 條：供應商可以將我們的使用資料用於「任何目的」。<br/>   - 第 7.2 條：供應商可以以「便利」為由終止，僅需 7 天通知。<br/>   - 第 11 條：沒有任何保證，即使是基本的適銷性保證。<br/>\</findings><br/><br/>\<recommendations><br/>1. 拒絕此協議。對於企業來說，風險遠大於收益。<br/>2. 反提案：<br/>   - 賠償：限制為因我們違反協議而產生的索賠。<br/>   - 責任：直接損害上限為 12 個月費用；協商間接損害。<br/>   - 知識產權：對我們的修改/整合保有完全所有權。<br/>   - 資料使用：限制為僅用於服務改進。<br/>   - 終止：最少 90 天通知；僅因故終止。<br/>   - 保證：要求基本適銷性和適用性保證。<br/>3. 如果供應商拒絕，探索替代供應商或內部解決方案。<br/>4. 鑑於風險很高，聘請法律顧問進行合約談判。<br/>\</recommendations> |
  
</section>
